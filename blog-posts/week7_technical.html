What is SQL injection you ask?

To keep it simple, SQL injection is what happens when a malicious user (i.e. a jerk, mom, or hacker) inserts SQL commands into an SQL statement using web pages' input fields (like username or password fields)

In the case from XKCD comic, Robert's mom inserted a DROP TABLE command into the field where she was supposed to put her son's first name. She piggybacked the command on her son's name, along with some well-setup syntax that would fit in perfectly with how the school's website reads code.

Why is this such a problem?

Well, because smart,evil dudes/ladies can take advantage of these opportunities to A. Alter the tables/fields/data in a database, B. Pull all the data for their own purposes, like passwords, credit card information, etc., C. Just dick around or make a point.

Assuming the malicious party knows how the website's SQL code interprets user input, they can insert commands like "SELECT * FROM bank_info WHERE 1=1;", and get all of the information from that table since 1=1 always comes out to True for any field. Next thing you know, all the bank info you had on that website's table is up for grabs somewhere on the internet.

What can you do to combat this issue?

Use SQL parameters. I'm not an expert on this topic, but I can break down how it works. SQL parameters are values that are added to an SQL query ONLY when it's time for the code to be executed. To quote from W3Schools' EXCELLENT tutorial on SQL Injection, "The SQL engine checks each parameter to ensure that it is correct for it's column and are treated literally, and not as part of the SQL to be executed".

To define something as a parameter you would use an @ sign in the line for WHERE. So it would be like...

txtWarriorName = getRequestString("WarriorName");

txtSQL = "SELECT * FROM warriors WHERE WarriorName = @0";

db.Execute(txtSQL,txtWarriorName);

Does this make sense? You're sort of compartmentalizing your data inputs, and keeping them protected unti it's time to execute.
